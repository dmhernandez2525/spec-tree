Spec Tree serves as the **upstream specification provider** for AI coding tools. This document details every export format, ensuring AI agents receive contextually rich, properly structured specifications.

**Key Insight:** Every AI coding tool performs dramatically better with structured specifications. Spec Tree solves the "garbage in, garbage out" problem.

## 1. Cursor Rules Export

### Overview
Cursor uses `.cursor/rules/` directory with MDC (Markdown Configuration) files to provide project context to the AI.

### Output Location
```
.cursor/
└── rules/
    ├── project.mdc        # Global project rules
    ├── frontend.mdc       # Frontend-specific rules
    ├── backend.mdc        # Backend-specific rules
    └── feature-*.mdc      # Feature-specific rules
```

### Format Specification

```markdown
---
title: [Project/Feature] Rules
description: AI coding rules generated by Spec Tree
globs: ["src/**/*"]  # Optional: scope to specific files
alwaysApply: false   # Optional: apply to all conversations
---

# Project: [Project Name]

## Tech Stack
- **Framework**: Next.js 14 (App Router)
- **Language**: TypeScript (strict mode)
- **Styling**: Tailwind CSS
- **State**: Zustand + TanStack Query
- **Forms**: React Hook Form + Zod
- **UI**: Radix UI primitives

## Code Style
- Use `cn()` utility for conditional Tailwind classes
- Components in PascalCase: `MyComponent.tsx`
- Hooks with `use` prefix: `useMyHook.ts`
- Types with `interface` keyword (not `type` for objects)

## Forbidden Patterns
- NO `any` types
- NO `@ts-ignore`
- NO `console.log` (use logger service)
- NO empty catch blocks

## File Structure
```
src/
├── app/              # Next.js App Router pages
├── components/       # React components
│   ├── ui/          # Reusable UI components
│   └── features/    # Feature-specific components
├── hooks/           # Custom React hooks
├── lib/             # Utilities and services
├── types/           # TypeScript type definitions
└── store/           # Zustand stores
```

## Current Feature Context
[Feature specification from work item]

### User Story
As a [user type], I want [capability] so that [benefit].

### Acceptance Criteria
- [ ] [AC-1]
- [ ] [AC-2]
- [ ] [AC-3]

### Files to Create/Modify
- `src/components/Feature.tsx` - Main component
- `src/hooks/useFeature.ts` - Custom hook
- `src/__tests__/Feature.test.tsx` - Test file

## Reference Patterns
- Button pattern: `src/components/ui/button.tsx`
- Form pattern: `src/components/forms/LoginForm.tsx`
- API pattern: `src/lib/api/client.ts`
```

### Context Aggregation

Cursor export aggregates:
1. Project-level tech stack and conventions
2. Parent context (Epic → Feature → Story)
3. Acceptance criteria from work item
4. Code patterns from referenced files

## 2. GitHub Copilot Export

### Overview
GitHub Copilot uses two formats:
1. `copilot-instructions.md` for project-level context
2. WRAP format for individual issues/tasks

### Output Location
```
.github/
└── copilot-instructions.md
```

### copilot-instructions.md Format

```markdown
# Project: [Project Name]

## Tech Stack
- Next.js 14 with App Router
- TypeScript in strict mode
- Tailwind CSS for styling
- Zustand for state management
- TanStack Query for server state

## Code Conventions
- All components must be typed with interfaces
- Use `cn()` for conditional class names
- Prefer Server Components where possible
- Use `useCallback` for event handlers passed to children

## Patterns
- API calls: Use the pattern in `src/lib/api/client.ts`
- Forms: Use React Hook Form with Zod validation
- Error handling: Always use try/catch with logger

## Current Sprint Goals
[From Epic/Feature context]
```

### WRAP Format (Issue/Task)

WRAP = **W**hat, **R**equirements, **A**ctual files, **P**atterns

```markdown
## What
[Clear description of the desired outcome from user story]

Implement a date range picker component that allows users to filter orders by date. The component should support preset ranges (Today, Last 7 days, Last 30 days) and custom date selection.

## Requirements
- [ ] Display preset date range options
- [ ] Support custom date range selection
- [ ] Validate that end date is after start date
- [ ] Sync selected range with URL parameters
- [ ] Show selected range in human-readable format
- [ ] Integrate with existing order filters

## Actual files
- `src/components/orders/DateRangePicker.tsx` - Main component
- `src/hooks/useDateFilter.ts` - Filter state hook
- `src/components/orders/OrderFilters.tsx` - Integration point
- `src/__tests__/DateRangePicker.test.tsx` - Tests

## Patterns
Follow the pattern established in:
- `src/components/ui/calendar.tsx` for calendar UI
- `src/components/ui/popover.tsx` for dropdown behavior
- `src/hooks/useUrlParams.ts` for URL sync

Use `date-fns` for date manipulation (already installed).
Use `cn()` for conditional Tailwind classes.
```

## 3. Devin Task Export

### Overview
Devin works best with atomic, 4-8 hour task specifications with verifiable acceptance criteria and clear verification commands.

### Key Principles
- **Atomic Tasks**: Single, focused objective
- **Verifiable Criteria**: Testable outcomes
- **Clear Context**: Dependencies and blockers
- **Verification Commands**: How to confirm completion

### Format Specification

```markdown
## Task: [Specific, single-purpose title]

**Scope**: 4-6 hours estimated work
**Labels**: devin, frontend, component
**Branch**: `feat/F2.1.1-date-range-picker`

### Context
This task is part of the Order History feature (Epic E-001).
- **Depends on**: F1.2.3 (Order List Component) - COMPLETE
- **Blocks**: F2.1.5 (Order Export), F2.1.6 (Order Analytics)

### User Story
As an admin user, I want to filter orders by date range so that I can analyze sales for specific periods.

### Acceptance Criteria
- [ ] Component renders without console errors
- [ ] Preset options work: Today, Last 7 days, Last 30 days, This month
- [ ] Custom date range picker allows selecting start and end dates
- [ ] Validation prevents end date before start date
- [ ] Selected range syncs to URL parameters
- [ ] Component is accessible via keyboard navigation
- [ ] All tests pass

### Technical Details

**File to create:**
```
src/components/orders/DateRangePicker.tsx
```

**Interface:**
```typescript
interface DateRangePickerProps {
  value: DateRange | null;
  onChange: (range: DateRange) => void;
  presets?: DatePreset[];
  minDate?: Date;
  maxDate?: Date;
  className?: string;
}

interface DateRange {
  start: Date;
  end: Date;
}
```

**Follow pattern:**
`src/components/ui/calendar.tsx` for calendar rendering
`src/components/ui/popover.tsx` for dropdown behavior

### Playbook Reference
Use playbook `component-creation` for standard component structure:
1. Create component file with TypeScript interface
2. Implement component with accessibility
3. Create test file with basic tests
4. Update barrel exports

### Verification Commands
```bash
# Check for TypeScript errors
pnpm typecheck

# Run component tests
pnpm test -- --testPathPattern=DateRangePicker

# Check for lint errors
pnpm lint

# Build to verify no issues
pnpm build
```

### Resources
- Design mockup: [Figma link]
- Similar component: `src/components/ui/date-picker.tsx`
- Date library docs: https://date-fns.org/docs/
```

### Linear Integration

When exporting to Linear for Devin:
- Auto-apply `devin` label
- Set estimate based on scope
- Link to parent Epic/Feature
- Include verification commands in description

## 4. v0 UI Spec Export

### Overview
v0 by Vercel excels at generating React/Tailwind UI components. Exports should emphasize visual specifications, states, and responsive behavior.

### Format Specification

```markdown
# Component: [Component Name]

## Description
[Brief description of what the component does]

## Visual Specifications

### Container
- Border: `border border-gray-200 dark:border-gray-800`
- Background: `bg-white dark:bg-gray-950`
- Border radius: `rounded-lg`
- Padding: `p-4`
- Shadow: `shadow-sm`

### Typography
- Heading: `text-lg font-semibold text-gray-900 dark:text-gray-100`
- Body: `text-sm text-gray-600 dark:text-gray-400`
- Label: `text-xs font-medium text-gray-500 dark:text-gray-500`

### Spacing
- Gap between items: `gap-4`
- Section margin: `mt-6`
- Button spacing: `space-x-2`

### Colors
- Primary action: `bg-blue-600 hover:bg-blue-700`
- Secondary action: `bg-gray-100 hover:bg-gray-200`
- Destructive: `bg-red-600 hover:bg-red-700`
- Success indicator: `text-green-600`
- Error indicator: `text-red-600`

## Component States

### Default State
- Show placeholder text in input
- Primary button enabled
- No error messages

### Loading State
- Show spinner in button
- Disable all inputs
- Show "Loading..." text

### Error State
- Red border on invalid input
- Error message below input
- Focus on first error field

### Success State
- Green checkmark icon
- Success message
- Redirect after 2 seconds

### Empty State
- Illustration or icon
- "No items yet" message
- Call-to-action button

## Responsive Behavior

### Desktop (>1024px)
- Two-column layout
- Side-by-side calendar months
- Full-width actions bar

### Tablet (768px - 1024px)
- Single column layout
- Stacked calendar months
- Full-width actions

### Mobile (<768px)
- Full-width popover
- Stacked form fields
- Bottom sheet for calendar
- Large touch targets (min 44px)

## Interactions

### Click Behavior
- Trigger: Opens popover
- Day: Selects date
- Preset: Applies immediately
- Apply: Closes popover, fires onChange

### Keyboard Navigation
- Tab: Move between focusable elements
- Enter/Space: Activate focused element
- Escape: Close popover
- Arrow keys: Navigate calendar

### Focus States
- Visible focus ring: `focus:ring-2 focus:ring-blue-500`
- Focus trap within popover

## Animations
- Popover: `animate-in fade-in-0 zoom-in-95`
- Selection: `transition-colors duration-150`
- Loading spinner: `animate-spin`

## Accessibility
- `role="dialog"` for popover
- `aria-label` on buttons
- `aria-selected` on dates
- Announce changes to screen readers
```

## 5. Windsurf Rules Export

### Overview
Windsurf uses XML-based rules files with specific tags for project context.

### Output Location
```
.windsurfrules
```

### Format Specification

```xml
<?xml version="1.0" encoding="UTF-8"?>
<windsurf_rules version="1.0">
  <project_context>
    <name>[Project Name]</name>
    <description>[Brief description]</description>
    <tech_stack>
      <framework>Next.js 14</framework>
      <language>TypeScript</language>
      <styling>Tailwind CSS</styling>
      <state>Zustand</state>
    </tech_stack>
  </project_context>

  <code_conventions>
    <convention>
      <name>Component Naming</name>
      <rule>Use PascalCase for component files</rule>
      <example>MyComponent.tsx</example>
    </convention>
    <convention>
      <name>Hook Naming</name>
      <rule>Prefix custom hooks with 'use'</rule>
      <example>useMyHook.ts</example>
    </convention>
    <convention>
      <name>Type Definitions</name>
      <rule>Use interface for object shapes</rule>
      <example>interface Props { ... }</example>
    </convention>
  </code_conventions>

  <forbidden_patterns>
    <pattern severity="critical">any</pattern>
    <pattern severity="critical">@ts-ignore</pattern>
    <pattern severity="critical">eslint-disable</pattern>
    <pattern severity="warning">console.log</pattern>
  </forbidden_patterns>

  <current_feature>
    <id>[Feature ID]</id>
    <title>[Feature Title]</title>
    <user_story>
      <as_a>[User type]</as_a>
      <i_want>[Capability]</i_want>
      <so_that>[Benefit]</so_that>
    </user_story>
    <requirements>
      <requirement id="1" priority="P0">[Requirement]</requirement>
      <requirement id="2" priority="P1">[Requirement]</requirement>
    </requirements>
    <files>
      <file action="create">src/components/Feature.tsx</file>
      <file action="modify">src/pages/index.tsx</file>
    </files>
  </current_feature>

  <patterns>
    <reference_file purpose="component">src/components/ui/button.tsx</reference_file>
    <reference_file purpose="hook">src/hooks/useAuth.ts</reference_file>
  </patterns>
</windsurf_rules>
```

## 7. MCP Server Real-Time Context

### Overview
The MCP (Model Context Protocol) server provides real-time access to Spec Tree data for AI tools.

### Tool Definitions

```typescript
// get_project_context
{
  name: "get_project_context",
  description: "Retrieve full project context including tech stack, conventions, and current work items",
  inputSchema: {
    type: "object",
    properties: {
      project_id: { type: "string" },
      include_children: { type: "boolean", default: true },
      depth: { type: "number", default: 3 }
    },
    required: ["project_id"]
  }
}

// get_task_context
{
  name: "get_task_context",
  description: "Get detailed context for a specific task including parent chain and acceptance criteria",
  inputSchema: {
    type: "object",
    properties: {
      task_id: { type: "string" },
      include_ancestors: { type: "boolean", default: true },
      include_patterns: { type: "boolean", default: true }
    },
    required: ["task_id"]
  }
}

// update_task_status
{
  name: "update_task_status",
  description: "Update the status of a task when work is completed",
  inputSchema: {
    type: "object",
    properties: {
      task_id: { type: "string" },
      status: { enum: ["in_progress", "completed", "blocked"] },
      notes: { type: "string" }

    required: ["task_id", "status"]

}

## 9. CodeRabbit Configuration Export

### Overview
CodeRabbit is an AI-powered code review tool that provides contextual PR feedback. Spec Tree can generate `.coderabbit.yaml` configuration files with path-specific review instructions derived from SDDs.

### Output Location
```
.coderabbit.yaml
```

### Format Specification

```yaml
# Generated by Spec Tree from project specifications
language: "en-US"

reviews:
  profile: "chill"  # or "assertive" based on project settings
  request_changes_workflow: false
  high_level_summary: true
  review_status: true

  # Path-specific instructions generated from SDD
  path_instructions:
    # Generated from Feature F2.1.1 (Date Range Picker)
    - path: "src/components/orders/DateRangePicker.tsx"
      instructions: |
        This component implements the Date Range Picker feature (F2.1.1).

        Acceptance Criteria to verify:
        - Display preset date range options (Today, Last 7 days, Last 30 days)
        - Support custom date range selection
        - Validate that end date is after start date
        - Sync selected range with URL parameters

        Technical requirements:
        - Use date-fns for date manipulation
        - Follow calendar.tsx pattern for UI
        - Ensure keyboard accessibility

        Related files: src/hooks/useDateFilter.ts

    # Generated from project coding standards
    - path: "**/*.tsx"
      instructions: |
        React component review checklist:
        - No `any` types - use proper TypeScript interfaces
        - Use `cn()` utility for conditional Tailwind classes
        - Hooks dependencies are correct (no missing deps)
        - Error boundaries for async operations
        - Accessibility: proper ARIA labels, keyboard navigation

    - path: "**/*.ts"
      instructions: |
        TypeScript review checklist:
        - No `@ts-ignore` or `eslint-disable` comments
        - All functions have explicit return types
        - Error handling uses try/catch with logger service
        - No console.log statements (use logger)

    # Generated from test specifications
    - path: "**/*.test.{ts,tsx}"
      instructions: |
        Test review checklist:
        - Tests describe behavior, not implementation
        - Edge cases covered (empty state, error state, loading)
        - Async operations properly awaited
        - No hardcoded timeouts (use waitFor)

  auto_review:
    enabled: true
    drafts: false
    base_branches:
      - main
      - develop

chat:
  auto_reply: true

# Ignore patterns from project configuration
path_filters:
  - "!**/*.lock"
  - "!**/node_modules/**"
  - "!**/.next/**"
  - "!**/dist/**"
```

### Context Aggregation for CodeRabbit

Spec Tree aggregates the following into CodeRabbit config:

1. **Project Coding Standards** → Global `path_instructions` for all files
2. **Feature Acceptance Criteria** → File-specific review instructions
3. **Test Specifications** → Test file review rules
4. **Architecture Patterns** → Reference to existing patterns
5. **Security Requirements** → Security-focused review rules

### PR Description Enhancement

When generating PR descriptions, Spec Tree includes context for CodeRabbit:

```markdown
